## What Makes This Different

Most anomaly detection projects download a clean CSV and run a model. We built something different:

| Standard Approach | This Project |
|---|---|
| Static CSV dataset | Live Kafka event stream |
| Pre-labeled anomalies | 8 types of injected real-world corruption |
| Run ML on clean data | Run ML on raw dirty data intentionally |
| Manual steps between stages | Single command pipeline automation |
| Static dashboard snapshot | Power BI DirectQuery вҖ” live data on refresh |
| One data flow | Two parallel flows вҖ” ML on raw, insights on clean |
| No alerting | Threshold-based anomaly rate monitoring |

The producer injects **8 real-world anomaly types** that actually occur in production e-commerce systems вҖ” not random noise:

```
1. Negative prices          вҶ’ refunds logged incorrectly
2. Currency mismatch        вҶ’ paise vs rupees (price Г— 1000)
3. Bot traffic              вҶ’ user_id clusters, quantity = 0
4. Schema drift             вҶ’ prod_id vs product_id rename
5. Missing payment fields   вҶ’ mobile app version bug
6. Corrupt city encoding    вҶ’ DELHI / delhi / DEL / Dilli
7. Invalid event types      вҶ’ VIEW / AddToCart / BUY
8. Timestamp corruption     вҶ’ NaT / epoch zero / wrong format
```

This closed-loop design means we can actually **validate** the ML model вҖ” we know exactly what we injected and can verify what was found.

## ML Model вҖ” Why IsolationForest

We had zero labeled data вҖ” no column marking which rows are anomalies. This rules out supervised approaches. We chose IsolationForest for specific reasons:

```
Requirement                    IsolationForest    DBSCAN    LOF
в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
No labeled data needed              вң“               вң“        вң“
Works on 7 features together        вң“               вң—        вң—
Fast on 150,000 rows                вң“               вң—        вң—
Scores new incoming data            вң“               вң—        вң—
Handles mixed anomaly types         вң“               вң—        вң—
```

The model detects anomalies by measuring how few random splits are needed to isolate a data point. Anomalies вҖ” being rare and different вҖ” get isolated in fewer splits than normal transactions surrounded by similar data.

```python
    # Power BI alert card turns red
```

This shifts the system from passive detection to **active monitoring** вҖ” the difference between finding bad rows and knowing when the entire pipeline is under stress.
